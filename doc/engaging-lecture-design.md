# 退屈しない講義スライドの設計ノウハウ

別の授業資料を作る際の引き継ぎ用ドキュメント。IP3200（情報処理）の可視化回の再構成（2026-06）で確立し、効果を確認した手法をまとめる。特定の授業に依存しない形で書く。

## 診断：スライドが退屈になる典型パターン

- **機能ツアー化** — ツールの機能を仕様書の順番で1個ずつ紹介する（plot → xlim → label → subplot …）。各スライドのデータがバラバラのおもちゃデータ（乱数、`x+5`）だと特に致命的。物語がなく、マニュアルの読み上げになる。
- **ネタバレ** — 驚きのある題材（例：アンスコムの四重奏）を、結論の図から先に見せてしまう。緊張感が生まれる前に答えを言っている。
- **受け身のリンク集・実例集** — 「いろいろあります」と見せるだけで、学生が手も口も動かさない。
- **教訓が callout 1個** — 本当は面白いネタ（例：グラフによる印象操作）が、本筋の脇の注意書き1個に圧縮されている。

## 装置（効果を確認した順に）

### 1. コールドオープン＋伏線回収

- 講義シリーズの**最終回の成果物（謎つき）を初回の冒頭で見せる**。「この謎は最終回に自分のコードで解く」と宣言する。
- 謎の答え（固有名詞・オチ）は最後まで伏せる。
- 中間の各回のまとめに「謎を解く日まであと N 回 — 今日の道具がその武器になる」という1行を入れる。毎回の学習内容が伏線回収の部品になる。
- 例：Wikipedia ページビューの「1日だけの大きな山」を可視化回の冒頭で見せ、API回で記事名と原因を明かす。

### 2. クイズ → めくり（予想させてから見せる）

- 統計量・主張・数字だけを先に見せ、「どうなると思う？」と**予想させてから**図をめくる。
- 「隣の人と予想」「手で描いてみる」を挟むと能動性が上がる。
- revealjs なら `::: {.callout-important .fragment}` や `::: {.fragment}` で同一スライド内のめくりが作れる。
- 例：アンスコムは「統計量が4つとも一致する表」→「どんな形か予想」→ 1個ずつめくり → 2×2並べが落ち。
- 例：「平均所得524万円。平均以下の世帯は何%？ 半分くらい？」→ めくると 62.2%。

### 3. 1セクション1データ（「1枚のグラフを育てる」）

- ツールの機能紹介パートは、**興味を引く問いを持った実データ1本**で貫く。機能の紹介順はそのままでよい。データだけ統一する。
- 「裸のプロット → 線 → 重ねて凡例 → ラベル・タイトル → 軸の範囲」と、スライドごとに1改善ずつ「資料に載せられるレベル」まで育てる。最後に before/after を並べて見せる。
- 問いに答えて終わる（例：「東京は本当に暑くなったのか」→ 150年分の散布図で決着）。

### 4. 実データ主義＋出典明記

- おもちゃデータより**本物の公的データ**。学生の生活に近いものほど効く（気温、所得、人口、アクセス数）。
- 日本の公的データは政府標準利用規約（CC BY相当）で講義利用できる。出典は callout で明記する。
- 入手しやすい実データ源：
  - 気象庁「過去の気象データ検索」（etrn の HTML テーブル。スクレイピングで取れる）
  - 厚生労働省「国民生活基礎調査」（平均 vs 中央値の鉄板ネタ：平均524.2万/中央値405万/平均以下62.2%、2022年）
  - Wikimedia Pageviews API（リアルタイム性と謎解きに最適）
  - Datasaurus Dozen（CC公開。「統計量が同じなのに恐竜」、アンスコムの現代版）

### 5. 「騙されない側になる」パート

- 描く技術の裏面として**読む・見抜く技術**を独立セクションにする。学生の食いつきが良い。
- クイズ形式：「このグラフ、何かおかしい。どこ？」→ 種明かし。
- ルールを宣言すると緊張感が出る：「これから見せるグラフに**ウソのデータは1つもない**」。
- 鉄板ネタ：y軸切断（棒グラフは0から）、期間チェリーピック（実データで「都合のいい12年」を切り取ると逆トレンドに見せられる）、二軸、3D円グラフ。
- 捏造画像を使わず、**本物のデータから自分で「騙しグラフ」を作る**と権利問題もなく、種明かしで同じデータの正直版を並べられる。

### 6. 受け身コンテンツの能動化

- リンク集・ギャラリーには小さなタスクを付ける：「1つ選んで隣の人に30秒で説明（何のデータ？軸は何？何が見えた？）」。

### 7. ツール重複は「描き比べ」として明示する

- 2つのツールで同じ種類の図を教えるとき、暗黙の繰り返しに見せず、冒頭で「**同じ図を別の道具で作って得意・不得意を体感する回**」と宣言する。
- 機能の振り分けはツールの強みに従う（例：回せない静止画ツールから3Dを外し、回せるツール側に置く）。

## 演習の設計

- **馴染みのある題材をウォーミングアップに** — 数学で見慣れた関数（二次関数＋平方完成で検算）などは、新しい道具の練習として心理的コストが低い。本編の早い位置に置く。
- **演習にも驚きを仕込む** — プロットすると絵が出る データ（Datasaurus の星など）は「解くと報酬がある」演習になる。
- 演習はその時点で習った道具だけで解けるように調整する（後で習う機能を要求しない）。

## 作業上の注意

- **題材の候補は、先に実データで検証してから採用する。直感は外れる。** 例：「季節性がありそう」という直感で挙がった富士山・軽井沢・尾瀬は、Wikipediaページビューの実測ではスパイクが弱く（最大値が中央値の3〜6倍）ボツ。代わりに行事ページ（恵方巻1521倍、七夕310倍、土用の丑の日189倍）が圧勝だった。富士山に想定した「週末効果」も実測では逆（土曜が最低）。候補は **スパイク倍率＝最大値/中央値** で機械的にランキングし、図にして形を見てから選ぶ。
- 講義スライドの図は、ネタバレ防止の構成（統計量→予想→図）に合わせて**スライドの順番から設計**する。図が先にあるとつい先に見せたくなる。
- スライド内で見せるだけの図は `#| echo: false`、学生がコピペする想定のコードは echo: true（ファイルパスは学生のローカル想定に合わせ、レンダリング用には echo: false の別チャンクを使う）。
- データ取得・統計量はその場で計算して検証してから本文に書く（記憶で数字を書かない）。
- 注釈付きの図は一度画像に出力して目視確認する（ラベルの見切れ等）。
